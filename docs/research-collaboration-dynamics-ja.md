# Toban を用いた協働ダイナミクス研究 — 検討メモ

本資料は、シビックテックコミュニティおよび多様な主体が参加するプログラムにおける「人々の協働のダイナミズム」を、Toban で取得・index されているデータを一次資料として研究するための、初期検討メモです。共同研究者との議論のたたき台として作成しています。

---

## 1. Toban とは何か

Toban は、**役割に基づく貢献記録と報酬分配のためのオープンソース・システム**です。オープンソース開発、共創プロジェクト、ボランティア・コミュニティなど「参加者が流動的に入れ替わる協働」を対象としています。

### 解こうとしている課題

1. **貢献の可視化** — 誰が何に貢献したかを、ピア間の承認として記録する
2. **役割に応じた報酬分配** — 役割と在任期間に応じた報酬の自動分配
3. **オンボーディングの整流化** — 役割階層を明示し、新規参加者が貢献しやすい導線を作る

### 採用している仕組み

- **Hats Protocol** — 役割(Role)を NFT として表現し、階層構造を持たせる
- **ThanksToken (ERC20)** — ピア間の「ありがとう」をトークンとして記録(mint/transfer)
- **FractionToken (ERC1155)** — 役割に紐づく分配持分を表現
- **Splits Protocol** — 報酬の自動分配
- デプロイ先: Ethereum Sepolia(テスト)、Base(本番)
- Namestone / ENS によるアドレス → 人名の解決

## 2. 研究に使える一次データ(subgraph で index 済み)

Goldsky 上の GraphQL エンドポイントから以下の粒度でクエリできます。すべて**タイムスタンプ付き**で、**workspace(プログラム)単位**に整理されています。

| エンティティ | 主なフィールド | 社会学的な意味 |
|---|---|---|
| `Workspace` | creator, owner, 各 Hat ID | プログラム/プロジェクトの単位 — **横断比較のユニット** |
| `MintThanksToken` | from, to, amount, data, blockTimestamp | **ピア承認イベント**(誰が誰にいつ感謝を贈ったか)。data に理由情報が載る設計 |
| `TransferThanksToken` | from, to, amount, blockTimestamp | 承認トークンの二次流通(譲渡・集約) |
| `BalanceOfThanksToken` | owner, balance, updatedAt | 承認残高のスナップショット |
| `AmountOfMintThanksToken` | sender, amount(累積) | 送り手の総 mint 量 — 承認**発信**の強度 |
| `InitializedFractionToken` | hatId, wearer, blockTimestamp | **役割への就任イベント**(誰がいつどの役になったか) |
| `TransferFractionToken` | from, to, tokenId, amount | 役割持分の移動 = 実質的な責任の再配分 |
| `BalanceOfFractionToken` | owner, hatId, balance, updatedAt | 各人の役割持分スナップショット |
| HatsTimeFrameModule(コントラクト) | 在任期間 | 役割の tenure |

### まとめると Toban は

> **タイムスタンプ付き・役割情報付きの、ピア承認ネットワーク**を、プログラム横断で取得できるデータソース

です。ピア承認のデータセットは既存研究でも希少(Slack の reactji 研究、GitHub の thanks bot など散発的)であり、その中でも Toban は「**役割・在任期間・報酬との接続**」までが同一ID空間で取れる点で独特です。

## 3. 研究として成立しうる問い(4方向)

### 方向 A. ネットワーク構造と「協働のかたち」

- Thanks-token グラフの構造的特徴(密度・相互性・クラスタリング・中心性)は**プログラムごとにどう異なるか**
- **仲介者(broker)** は誰か — サブグループを橋渡しする人の属性と役割
- 時間発展: 立ち上げ期 → 安定期 → 収束期でネットワーク構造はどう変化するか
- **媒体間比較**: Thanks グラフ ↔ GitHub(共同編集・レビュー) ↔ Slack(メンション・リアクション) ↔ Notion(共同編集)。同じ人の協働でも、媒体ごとに異なる構造が現れるはず。この差分こそが「何を Toban が見せ、何を見落とすか」を示す

### 方向 B. 役割・キャリア・定着

- 新規参加者の onboarding 経路: どの役から入り、どう遷移するか(sequence analysis / survival analysis)
- 役割の在任期間と承認流入量の相関
- **離脱の予兆**: 承認の出入りが減ってから何週間で役割から抜けるか(early warning モデル)

### 方向 C. 分配の公正と認識のズレ

- FractionToken 配分の集中度(Gini 係数)と、当事者の**体感的公正感**の乖離
- **「見えない労働」の可視性**: ファシリテーション、感情労働、場のケア、翻訳、調整——これらは Thanks に現れているか、それとも沈んでいるか
- ジェンダー・所属組織・地域などの属性別に承認の偏りが存在するか

### 方向 D. Toban 導入そのものの介入効果(メタ研究)

- 導入前後で協働のされ方が変わったか(pre/post デザイン、または導入/未導入プログラムの比較)
- **観測者効果**: 承認がトークン化されることで、承認行為自体がパフォーマティブ化していないか

## 4. 混合研究デザインの提案

Toban のデータは強力ですが、次の問いには答えられません:

- 承認の**意味**(なぜその人に mint したのか)
- **mint されなかった貢献**(オフチェーンに沈んだ労働)
- 参加の**動機**と**断念の理由**
- コミュニティの**境界認識**(当事者にとっての in/out は Thanks グラフと一致するか)
- Toban の UI が行動をどう誘導しているか

以下の質的調査でこれを補完します。

### 推奨デザイン

1. **事例比較(3〜4 workspaces)**
   性質の異なるプログラムをサンプリング(例: 自治体協働型 / オープンデータ型 / 短期ハッカソン型)

2. **ネットワーク指標で対象者抽出 → エゴネットワーク・インタビュー**
   高中心性・高仲介(broker)・周縁・離脱の4象限から、各 5〜8 名

3. **参加者日誌 / ワークショップ・エスノグラフィー**
   1プログラムを深掘り(3ヶ月程度の観察)

4. **アンケート(アドレス紐付き・同意ベース)**
   心理的安全性・手続き的公正・所属感・動機(SDT)などの既存尺度を利用

## 5. 倫理的配慮

- **同意と再識別リスク**: ENS / Namestone に紐づいたアドレスは実質的に識別子。研究利用について、コミュニティ全体への事前説明とオプトアウト機会が必須
- **トークン量 ≠ 貢献量**: mint 上限、文化差、スパム的 mint、ボットの存在に注意
- **ホーソン効果 / パフォーマティビティ**: 「研究されている」「可視化されている」こと自体が協働行動を歪める可能性。これは研究対象としても扱うべき
- **データ保存期間とアクセス制御**: オンチェーンである以上、公開データではあるが、研究目的の集約・分析結果の公開範囲は別途設計する

## 6. 次に議論したい論点

- リソース・期間・アクセス可能なコミュニティから、どの方向(A〜D)から着手するのが現実的か
- 比較ケースに含めたい workspace(CfJ 内外)はあるか
- 同意取得の進め方 — コミュニティマネージャとの連携体制
- 定量分析の実装: 本リポジトリの subgraph から直接 GraphQL で取得するか、スナップショットを別途保存するか
- 研究成果のアウトプット先(学術論文 / コミュニティ向けレポート / Toban 設計への還元)
